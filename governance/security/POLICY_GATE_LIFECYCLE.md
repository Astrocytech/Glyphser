# Policy/Gate Lifecycle

## Purpose
Define the required lifecycle for security policy and gate changes so updates are reviewable, signed, and reversible.

## Lifecycle
1. Propose
- Open a change with threat/control rationale and impacted workflows.
- Include policy diffs in `governance/security/*.json` and test updates in `tests/security/`.

2. Approve
- Obtain required reviewers per `governance/security/review_policy.json`.
- Include change ticket metadata when security-critical paths are touched.

3. Deploy
- Merge with all security lanes passing in CI (`security-matrix`, super-gate, signature gates).
- Ensure evidence artifacts for changed controls are produced under run-scoped evidence roots.

4. Rollback
- Use release rollback controls (`release_rollback_provenance_gate`) and documented emergency workflow paths.
- Record rollback decision, run ID, and impacted controls in `governance/security/SECURITY_CHANGELOG.md`.

5. Deprecate
- Mark old control/policy as deprecated with a target removal date.
- Add replacement control mapping and migration evidence before removal.

## Required Artifacts
- Signed policy files (`*.json.sig`) and signed policy manifest.
- Updated runbook entries in `GATE_RUNBOOK_INDEX.md`.
- Tests proving negative-path behavior for new/changed gate logic.
- Evidence reports in `evidence/security/` generated by CI/release/maintenance lanes.
